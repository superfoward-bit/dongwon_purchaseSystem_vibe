# DB 설계 (공통) — 문서 상태전이 모델 (전진 + 역방향 되돌리기)

> 핵심 요구: 모든 업무 프로세스는 **앞으로 진행(전진)**뿐 아니라, 문제가 생기면 **이전 단계로 되돌리기(역방향)**가 가능해야 한다.
> 이 모델은 PR/RX/AU/CT/PO/GR/CL/품목단가/협력사 등 STS를 가진 **모든 문서에 공통 적용**한다.

---

## 1. 상태전이 액션 표준 (Transition Actions)

| 액션 | 코드 | 방향 | 설명 | 주체 |
|------|------|------|------|------|
| 전진 | `FORWARD` | → | 다음 단계로 진행 (상신/승인/완료 등) | 담당/결재자 |
| 반려 | `REJECT` | ← | 결재·검수자가 **이전 단계로 되돌림** + 사유 | 검토/결재/검수자 |
| 반송 | `RETURN` | ← | 다음 담당자가 **이전 담당자에게** 되돌림 | 후속 담당자 |
| 회수 | `RECALL` | ← | 상신자가 **진행중인 자기 문서를 회수** (결재 전/중) | 기안자 |
| 취소 | `CANCEL` | ✕ | 문서 자체를 취소(종료) | 담당/관리자 |
| 재상정 | `REOPEN` | ↻ | 반려/회수된 문서를 수정 후 **다시 진행** | 기안자 |
| 강제이동 | `FORCE` | ↔ | 관리자가 권한으로 임의 단계 이동(예외처리) | 관리자 |

> 모든 역방향 액션은 **사유(RSN) 필수 입력**, 알림(메일/SMS) 발송, 변경이력 기록.

---

## 2. 전이 규칙 관리 (하드코딩 금지)

상태전이는 코드에 박지 않고 **전이맵 테이블**로 관리 → 문서유형별로 "어느 상태에서 어떤 액션으로 어느 상태로 갈 수 있는지" 데이터로 정의.

### CM_STATUS_FLOW (상태전이 규칙)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| ID 🔑 | NUMBER | 대리키 |
| DOC_TYP ⭐ | VARCHAR2(18) | 문서유형(PR/RFX/AUCTION/PO/GR/CT/CL…) |
| FROM_STS ⭐ | VARCHAR2(18) | 현재 상태 |
| ACTION ⭐ | VARCHAR2(18) | 액션(FORWARD/REJECT/RECALL/RETURN/CANCEL/REOPEN/FORCE) |
| TO_STS | VARCHAR2(18) | 전이 후 상태 |
| ALLOW_ROLE | VARCHAR2(100) | 허용 역할/권한 |
| RSN_REQ_YN | CHAR(1) | 사유 필수 여부 |
| NOTI_YN | CHAR(1) | 알림 발송 여부 |
| SORT_NO | NUMBER | 버튼 노출 순서 |
> UK: DOC_TYP+FROM_STS+ACTION. 화면은 이 테이블을 조회해 **현재 상태에서 가능한 액션 버튼만** 동적으로 노출.

---

## 3. 상태 이력 (되돌린 기록까지 전부 추적)

### LOG_DOC_STATUS_HIS (문서 상태이력) ★공통
| 컬럼 | 타입 | 설명 |
|------|------|------|
| ID 🔑 | NUMBER | 대리키 |
| DOC_TYP | VARCHAR2(18) | 문서유형 |
| DOC_ID | NUMBER | 문서 ID |
| DOC_NO | VARCHAR2(30) | 문서번호(조회용) |
| SEQ | NUMBER | 전이 순번 |
| FROM_STS | VARCHAR2(18) | 이전 상태 |
| TO_STS | VARCHAR2(18) | 변경 상태 |
| ACTION | VARCHAR2(18) | 수행 액션 |
| STEP_NM | VARCHAR2(100) | 단계명(표시용) |
| ACTOR_ID | VARCHAR2(18) | 처리자 |
| ACTOR_ROLE | VARCHAR2(50) | 처리자 역할 |
| ACTION_DT | TIMESTAMP | 처리일시 |
| RSN | VARCHAR2(1000) | **사유(반려/회수/취소 시 필수)** |

> 문서 화면의 "진행이력" 탭 = 이 테이블 조회. 정방향·역방향 모두 시간순으로 표시.

---

## 4. 도메인별 역방향(되돌리기) 적용 예시

| 문서 | 전진 흐름 | 되돌리기(역방향) 가능 지점 |
|------|----------|--------------------------|
| **PR 구매요청** | 작성→상신→결재→소싱→계약 | 결재중→`반려`→작성 · 상신→`회수`→작성 · 소싱중→`취소` · 결재완료→`재작성`(반려시) |
| **RFx 견적** | 작성→공고→진행→평가→선정 | 평가중→`재견적(REOPEN)`→공고 · 공고→`취소` · 진행→`반려`→작성 |
| **역경매** | 작성→공고→입찰→낙찰 | 공고→`취소` · 입찰→`유찰/재공고` · 낙찰→`취소` |
| **단가계약** | 작성→결재→계약 | 결재중→`반려`→작성 · 계약→`해지/변경` |
| **PO 발주** | 작성→결재→발주완료→입고 | 결재중→`반려`→작성 · 발주완료→`변경(수정판수)`/`취소` · 입고전 회수 |
| **GR 입고검수** | 작성→검수→결재→마감 | 검수 `불합격`→발주 `반송`/재입고 · 검수중→`반려` · 마감전 취소 |
| **품목단가/인도가** | 변경→임시→결재→SAP반영 | 결재중→`반려`→임시 · SAP실패(IF_STATUS=F)→`재처리` · 적용전 `회수` |
| **협력사 등록/평가** | 신청→결재→등록 | 결재중→`반려`→신청 · 보완요청(`RETURN`)→재신청 |

> SAP 연계 단계(IF_STATUS) 실패 시: 자동 `재처리(재전송)` 또는 수동 되돌리기 → 결재단계로 반려.

---

## 5. 구현 가이드 (요약)
- 서비스 공통 `StatusTransitionService.move(docTyp, docId, action, rsn)` → ① CM_STATUS_FLOW로 유효성·권한 검증 ② 문서 STS 갱신 ③ LOG_DOC_STATUS_HIS 기록 ④ 알림 ⑤ 변경이력(LOG_CHANGE).
- 화면: 현재 STS 기준으로 가능한 액션 버튼 동적 렌더(전진/반려/회수/취소).
- 역방향 시 후속 데이터 정리 규칙 정의(예: 발주 취소 시 연결 입고 존재하면 차단).
